Entity (เอนทิตี)

สิ่งที่ต้องการเก็บข้อมูลในระบบ — มีตัวตนหรือแนวคิดที่ชัดเจน

Strong Entity (เอนทิตีหลัก)
STRONG ENTITY
สิ่งที่มีตัวตนอิสระ มี Primary Key เป็นของตัวเอง วาดด้วยสี่เหลี่ยมเส้นเดี่ยว
✦ ตัวอย่าง: นักศึกษา, สินค้า, พนักงาน, ลูกค้า, วิชา
Weak Entity (เอนทิตีอ่อน)
WEAK ENTITY
ไม่มี Primary Key เป็นของตัวเอง ต้องอาศัย Strong Entity ระบุตัวตน วาดด้วยสี่เหลี่ยมเส้นคู่
✦ ตัวอย่าง: ผู้ติดตาม (ของพนักงาน), รายการสั่งซื้อ, ห้องพัก (ของอาคาร)
💡 หลักการเลือก Entity
สิ่งที่จะเป็น Entity ต้องผ่านเงื่อนไข: (1) มีตัวตนชัดเจน (2) เก็บข้อมูลได้มากกว่า 1 Attribute (3) มีความสัมพันธ์กับ Entity อื่น
✓ Entity: นักศึกษา, สินค้า, แผนก    ✗ ไม่ใช่ Entity: อายุ, ราคา, ชื่อ (เหล่านี้คือ Attribute)
🟢

Attribute (แอตทริบิวต์)

คุณสมบัติหรือข้อมูลที่ต้องการเก็บของ Entity

Simple Attribute
SIMPLE / ATOMIC
Attribute ธรรมดา แบ่งย่อยไม่ได้อีก วาดด้วยวงรีเส้นเดี่ยว
✦ ชื่อ, อายุ, เพศ, ราคา, วันที่
Key Attribute (Primary Key)
KEY ATTRIBUTE
ระบุตัวตนของ Entity ได้ไม่ซ้ำกัน วาดวงรีเส้นเดี่ยว แต่ขีดเส้นใต้ชื่อ Attribute
✦ student_id, emp_id, product_code
Partial Key
PARTIAL KEY / DISCRIMINATOR
Key ของ Weak Entity ที่ไม่สามารถระบุได้เพียงลำพัง ต้องใช้คู่กับ Key ของ Strong Entity ขีดเส้นใต้เส้นประ
✦ เลขห้อง (ของอาคาร), ลำดับที่ (ของออเดอร์)
Derived Attribute
DERIVED ATTRIBUTE
คำนวณได้จาก Attribute อื่น ไม่ต้องเก็บจริงในฐานข้อมูล วาดวงรีเส้นประ
✦ อายุ (จากวันเกิด), ราคารวม (จากราคา×จำนวน)
Multi-valued Attribute
MULTI-VALUED
เก็บค่าได้หลายค่า วาดวงรีซ้อนกัน 2 ชั้น
✦ เบอร์โทร (หลายเบอร์), ภาษาที่พูดได้, ทักษะ
Composite Attribute
COMPOSITE ATTRIBUTE
Attribute ที่แบ่งย่อยได้ต่ออีก มี Sub-Attribute ข้างใต้ เชื่อมด้วยเส้น
✦ ที่อยู่ (บ้านเลขที่, ถนน, จังหวัด), ชื่อเต็ม (ชื่อ, นามสกุล)
สัญลักษณ์ ประเภท ลักษณะเส้น สถานการณ์ที่ใช้
Simple วงรีเส้นเดี่ยว ข้อมูลทั่วไป แบ่งย่อยไม่ได้
Key วงรี + ขีดเส้นใต้ Primary Key ระบุตัวตนได้
Partial Key วงรี + เส้นประใต้ Key ของ Weak Entity
Derived วงรีเส้นประ คำนวณได้จากค่าอื่น
Multi-valued วงรีซ้อน 2 ชั้น มีค่าได้หลายค่า
🔸

Relationship (ความสัมพันธ์)

การเชื่อมโยงระหว่าง Entity ตั้งแต่ 2 ตัวขึ้นไป

Regular Relationship
RELATIONSHIP
ความสัมพันธ์ระหว่าง Strong Entity วาดด้วยรูปเพชรเส้นเดี่ยว
✦ ลงทะเบียน, ซื้อ, จัดการ, สังกัด
Identifying Relationship
WEAK RELATIONSHIP
ความสัมพันธ์ระหว่าง Weak Entity กับ Strong Entity ที่เป็นเจ้าของ วาดด้วยรูปเพชรเส้นคู่
✦ เป็นของ, สังกัด (ของ Weak Entity)
CARDINALITY RATIO — ตัวเลขบนเส้น
1 : 1
พนักงาน 1 คน มีรหัสบัตร 1 ใบ — ทั้งสองฝั่งมีได้แค่ 1 ตัว
1 : N
แผนก 1 แผนก มีพนักงานได้หลายคน แต่พนักงาน 1 คนสังกัดแผนกเดียว
M : N
นักศึกษา 1 คน ลงวิชาได้หลายวิชา / วิชา 1 วิชา มีนักศึกษาหลายคน
🟣

Weak Entity (เอนทิตีอ่อน)

Entity ที่ไม่มี Primary Key เป็นของตัวเอง ต้องอาศัย Strong Entity

📦 ตัวอย่าง: ระบบพนักงานและผู้ติดตาม
Weak Entity Identifying Rel Partial Key
อ่านอย่างไร?
เส้นคู่รอบ "ผู้ติดตาม" = Weak Entity ไม่มี PK อิสระ
เพชรคู่ "มี" = Identifying Relationship
"ชื่อ" ขีดเส้นประใต้ = Partial Key ต้องใช้คู่กับ emp_id
เส้นคู่ฝั่งซ้าย = Total Participation (พนักงานทุกคนต้องมี Weak Entity)
1:N = พนักงาน 1 คน มีผู้ติดตามได้หลายคน
〰️

Participation Constraint (เส้นเชื่อม)

บอกว่า Entity ทุกตัวต้องมีความสัมพันธ์หรือไม่

Total Participation
เส้นคู่ (Double Line)
ทุกอินสแตนซ์ของ Entity ต้องมีความสัมพันธ์
ไม่มีตัวที่ไม่มีคู่ได้ = Mandatory
✅ พนักงานทุกคนต้องสังกัดแผนก
✅ Billทุกใบต้องมีการสั่งซื้อ
Partial Participation
เส้นเดี่ยว (Single Line)
บางอินสแตนซ์ของ Entity อาจไม่มีความสัมพันธ์
มีตัวที่ไม่มีคู่ได้ = Optional
⚪ พนักงานบางคนอาจไม่ดูแลโครงการ
⚪ ลูกค้าบางรายอาจยังไม่เคยสั่งซื้อ
🔍 เปรียบเทียบ Total vs Partial บน Diagram

Specialization / Generalization

ความสัมพันธ์แบบ Is-a ระหว่าง Superclass และ Subclass

🏦 ตัวอย่าง: ระบบบัญชีธนาคาร
Disjoint Total
อ่าน Specialization อย่างไร?
วงกลม "d" = Disjoint บัญชี 1 บัญชีเป็นได้แค่ประเภทเดียว
วงกลม "o" = Overlapping เป็นได้หลายประเภทพร้อมกัน
เส้นคู่จาก Superclass → วงกลม = Total Specialization (ต้องจัดอยู่ใน Subclass)
เส้นเดี่ยว = Partial Specialization (บางตัวอาจไม่อยู่ใน Subclass)
Subclass รับ Attribute ทุกตัวจาก Superclass + มี Attribute เฉพาะของตัวเอง
Disjoint (d)
เป็นได้แค่ Subclass เดียว
บัญชีออมทรัพย์ ≠ กระแสรายวัน
Overlapping (o)
เป็นได้หลาย Subclass พร้อมกัน
คนเดียวเป็นทั้งอาจารย์และนักศึกษา
🗺️

ตัวอย่าง ER-Diagram เต็มระบบ

ระบบงานจริงที่ใช้สัญลักษณ์ครบทุกประเภท

🏢 ระบบ HR — พนักงาน + แผนก + โครงการ
Strong Entity Weak Entity Total Part. Partial Part. M:N Rel.
✦ สัญลักษณ์ที่ใช้ในแผนภาพนี้
เส้นคู่ (Total) ระหว่างพนักงานกับแผนก = พนักงานทุกคนต้องสังกัดแผนก
เส้นเดี่ยว (Partial) ระหว่างพนักงานกับโครงการ = บางคนอาจไม่ทำโครงการ
วงรีซ้อน "เบอร์โทร" = Multi-valued Attribute มีได้หลายเบอร์
เพชรคู่ + กล่องคู่ = Weak Entity "ผู้ติดตาม" กับ Identifying Relationship
วงรีเส้นประ (Derived) บน Relationship = Attribute ของ Relationship "ชม./สัปดาห์"
เส้นวนกลับ (Recursive) = พนักงานดูแลพนักงาน Role: Supervisor:Supervisee
✏️

วิธีวาด ER-Diagram (Step by Step)

กระบวนการวาดที่ถูกต้องตั้งแต่เริ่มต้น

1
อ่านโจทย์และขีดเส้นใต้ "คำนาม"
คำนามในโจทย์มักเป็น Entity ที่ต้องวาด เช่น "พนักงาน", "แผนก", "โครงการ" ขีดเส้นใต้ไว้ก่อน
เทคนิค: คำนามที่เก็บข้อมูลได้หลายรายการ = Entity / คำนามที่เป็นแค่ค่าเดียว = Attribute
2
ระบุ Attribute ของแต่ละ Entity
หาคุณสมบัติที่ต้องเก็บของแต่ละ Entity แล้วจัดประเภท: Simple, Key, Multi-valued, Derived, Composite
ต้องมีเสมอ: Key Attribute อย่างน้อย 1 ตัวต่อ 1 Strong Entity
3
ระบุ Relationship และตั้งชื่อ
หาว่า Entity คู่ไหนมีความสัมพันธ์กัน ตั้งชื่อ Relationship เป็นกริยา เช่น "สังกัด", "ลงทะเบียน", "จัดการ"
ถามว่า: "Entity A _____ Entity B" ช่องว่างคือชื่อ Relationship
4
กำหนด Cardinality (1:1, 1:N, M:N)
ถามทั้ง 2 ทิศ: "A 1 ตัว ไปกับ B กี่ตัว?" และ "B 1 ตัว ไปกับ A กี่ตัว?" เขียนตัวเลขบนเส้น
จำง่าย: ตัวเลขเขียนใกล้ Entity ที่ตัวเลขนั้นชี้ถึง
5
กำหนด Participation (Total / Partial)
ถามว่า "Entity ทุกตัวต้องมีความสัมพันธ์ไหม?" ถ้าใช่ = Total (เส้นคู่) ถ้าไม่ = Partial (เส้นเดี่ยว)
คำชี้บอก: "ทุก...", "ต้องมี..." = Total  |  "บางส่วน...", "อาจจะ..." = Partial
6
ตรวจสอบ Weak Entity และ Specialization
มี Entity ใดที่ไม่มี PK อิสระไหม? (= Weak Entity) มี Entity ที่แบ่งประเภทย่อยได้ไหม? (= Specialization)
Weak Entity: ถ้า Entity นั้นหายไปเมื่อ Strong Entity ถูกลบ = Weak Entity
7
วาดและจัดวางให้ชัดเจน
วาง Entity ให้ห่างพอ, เส้นไม่ตัดกันมากเกินไป, เขียนตัวเลข Cardinality ให้อ่านชัด, ไม่ลืม Attribute
เรียงลำดับ: Entity หลักตรงกลาง, Entity ที่เชื่อมกันมากวางใกล้กัน
⚡ เทคนิคและข้อผิดพลาดที่พบบ่อย
🚫
