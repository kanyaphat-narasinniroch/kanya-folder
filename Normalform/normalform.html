Normal Form

การทำ Normalization ฐานข้อมูล

1NF · 2NF · 3NF · BCNF

🗄️
Normalization คืออะไร? แนวคิดและหลักการพื้นฐาน
DEFINITION
Normalization คือกระบวนการออกแบบโครงสร้างตาราง (Table) ในฐานข้อมูลให้มีประสิทธิภาพ โดยลดความซ้ำซ้อนของข้อมูล และป้องกันปัญหาที่อาจเกิดขึ้นในการจัดการข้อมูล
🎯 เป้าหมายของ Normalization
GOAL 01
ลด Data Redundancy
ลดข้อมูลที่ซ้ำซ้อนในฐานข้อมูล ประหยัดพื้นที่จัดเก็บ
GOAL 02
ป้องกัน Anomaly
ป้องกันข้อผิดพลาดเมื่อเพิ่ม แก้ไข หรือลบข้อมูล
GOAL 03
Data Integrity
ทำให้ข้อมูลมีความสมบูรณ์และน่าเชื่อถือ
GOAL 04
ง่ายต่อการดูแล
โครงสร้างที่ดีทำให้แก้ไขและขยายระบบได้ง่ายขึ้น
⚠️ ปัญหา Anomaly ที่เกิดจากตารางไม่ดี
➕
Insert Anomaly
เพิ่มข้อมูลไม่ได้ ถ้าข้อมูลที่เชื่อมกันยังไม่มีในระบบ เช่น เพิ่มวิชาใหม่ไม่ได้ ถ้าไม่มีนักศึกษาลงทะเบียน
✏️
Update Anomaly
แก้ไขข้อมูลต้องแก้หลายแถว ถ้าลืมแก้บางแถว ข้อมูลจะไม่ตรงกัน
🗑️
Delete Anomaly
ลบข้อมูลหนึ่งแล้วสูญเสียข้อมูลอื่นที่ต้องการเก็บไว้ด้วย
🔑 แนวคิดสำคัญ: Functional Dependency (FD)
DEFINITION
A → B หมายความว่า "ถ้ารู้ค่า A แล้ว จะรู้ค่า B ได้เสมอ" หรือ B ขึ้นอยู่กับ A
รหัสนักศึกษา→ชื่อนักศึกษา(รู้รหัส → รู้ชื่อ)
รหัสวิชา→ชื่อวิชา, หน่วยกิต
รหัสไปรษณีย์→จังหวัด, อำเภอ
🗝️ ประเภทของ Key
ประเภท Keyคำอธิบายตัวอย่าง
Primary KeyAttribute ที่ระบุแต่ละแถวได้ไม่ซ้ำรหัสนักศึกษา
Candidate Keyตัวเลือกที่เป็น Primary Key ได้รหัสนักศึกษา, เลขบัตรประชาชน
Composite KeyPrimary Key ที่ประกอบหลาย attribute(รหัสนักศึกษา + รหัสวิชา)
Non-prime AttributeAttribute ที่ไม่ใช่ส่วนของ Candidate Keyชื่อ, เกรด, ที่อยู่
Superkeyชุด attribute ที่ระบุแถวได้ไม่ซ้ำ (อาจมี attribute เกิน)(รหัสนักศึกษา + ชื่อ)

📊
สรุปเปรียบเทียบ Normal Forms ทั้งหมด
📋 ตารางสรุปเปรียบเทียบ
Normal Formผ่านเมื่อปัญหาที่แก้เงื่อนไขหลัก
1NF ทุก cell มีค่าเดียว Repeating Groups Atomic Values
2NF ผ่าน 1NF + Full Dependency Partial Dependency ทุก attr ขึ้นกับ PK ทั้งก้อน
3NF ผ่าน 2NF + No Transitive Transitive Dependency ทุก attr ขึ้นกับ Key โดยตรง
BCNF ผ่าน 3NF + Superkey only Multi Candidate Key ทุก Determinant เป็น Superkey
1NF

ห้ามยัดหลายค่าในช่องเดียว

ทุกช่องในตารางต้องมีค่าเดียว (Atomic) ไม่มี repeating group เช่น "CS101, CS102" ในช่องเดียว

2NF

ทุก attribute ต้องขึ้นกับ Key ทั้งก้อน

ถ้า PK เป็น Composite Key ต้องไม่มี attribute ที่ขึ้นกับแค่บางส่วนของ PK (Partial Dependency)

3NF

ทุก attribute ต้องขึ้นกับ Key โดยตรง

ห้ามมี Non-prime Attribute ขึ้นกับ Non-prime Attribute อื่น (Transitive Dependency)

BCNF

ทุกตัวที่กำหนดค่าอื่นได้ต้องเป็น Superkey

เข้มงวดกว่า 3NF สำหรับกรณีที่มีหลาย Candidate Key ซ้อนทับกัน

🧠 สูตรจำแบบย่อ
1NF → ไม่มีหลายค่าในช่องเดียว
2NF → ไม่มี Partial Dependency (Key บางส่วน)
3NF → ไม่มี Transitive Dependency (ผ่านตัวกลาง)
BCNF → ทุก Determinant เป็น Superkey

📝 แบบฝึกหัดที่ 1 — ระบบร้านค้าออนไลน์

กำหนดตารางต่อไปนี้ ให้ทำ Normalization ให้ถึง 3NF พร้อมระบุ FD ที่พบ

คำสั่งซื้อ(รหัสคำสั่งซื้อ, รหัสลูกค้า, ชื่อลูกค้า, อีเมลลูกค้า,
              รหัสสินค้า, ชื่อสินค้า, ราคาต่อหน่วย, จำนวน, รวมเงิน,
              วันที่สั่งซื้อ, สถานะ)
  • ระบุ FD ทั้งหมดที่พบในตาราง
  • ตรวจสอบว่า 1NF ผ่านหรือไม่ ถ้าไม่ให้แก้ไข
  • หา Partial Dependency และแยกตารางให้ผ่าน 2NF
  • หา Transitive Dependency และแยกตารางให้ผ่าน 3NF
  • วาด Entity-Relationship Diagram (ERD) แสดงความสัมพันธ์ระหว่างตาราง
📝 แบบฝึกหัดที่ 2 — ระบบโรงพยาบาล

จากข้อมูลการรักษาพยาบาลต่อไปนี้

การรักษา(รหัสการรักษา, รหัสผู้ป่วย, ชื่อผู้ป่วย, วันเกิดผู้ป่วย,
            รหัสแพทย์, ชื่อแพทย์, ความเชี่ยวชาญแพทย์,
            รหัสห้อง, ชั้น, อาคาร, วันที่รักษา, การวินิจฉัย)
  • แยกตารางให้ผ่าน 3NF
  • ระบุ Primary Key และ Foreign Key ของทุกตาราง
  • อธิบายว่ามี Transitive Dependency อะไรบ้างในตารางเดิม
🎯 แบบฝึกหัดที่ 3 — BCNF Challenge

ตารางต่อไปนี้ผ่าน 3NF แต่อาจไม่ผ่าน BCNF ให้วิเคราะห์

ตารางสอน(นักศึกษา, วิชา, อาจารย์)

เงื่อนไข:
 • นักศึกษา 1 คน ลงวิชา 1 วิชาได้กับอาจารย์หลายคน
 • อาจารย์ 1 คน สอนได้เฉพาะวิชาเดียว
 • วิชาหนึ่งมีอาจารย์ผู้สอนได้หลายคน
  • ระบุ Candidate Key ทั้งหมดของตาราง
  • ระบุ FD ทั้งหมดและตรวจสอบว่า FD ใดละเมิด BCNF
  • แยกตารางให้ผ่าน BCNF
  • ตรวจสอบว่าการแยกตาราง lossless join หรือไม่
